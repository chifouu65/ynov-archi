# LIVRABLE SÉANCE 7 : Industrialisation

**Document :** Industrialisation & Versioning

## 1. Objectif : Préparer l'Industrialisation

L'objectif est de mettre en place les fondations pour un développement collaboratif robuste, garantissant la reproductibilité des builds et la gestion cohérente des versions.

## 2. Arborescence Logique et Build

Le projet suit le standard **Maven** strict.

### 2.1. Structure du Build

*   **Build Tool :** Apache Maven 3.8+
*   **JDK :** Java 21 (LTS)
*   **Encodage :** UTF-8

L'arborescence standard est respectée pour chaque module :
```text
module-name/
├── pom.xml
├── src/
│   ├── main/
│   │   ├── java/ (Sources)
│   │   └── resources/ (Configs)
│   └── test/
│       ├── java/ (Tests Unitaires)
│       └── resources/ (Configs Test)
```

## 3. Découpage en Modules (Figé)

Conformément à l'architecture définie en Séance 4, le découpage modulaire est figé comme suit :

1.  **`saas-kernel`** : Socle technique (zéro dépendance métier).
2.  **`saas-identity`** : Gestion des utilisateurs et droits.
3.  **`saas-ticketing`** : Cœur métier (Tickets).
4.  **`saas-billing`** : Facturation.
5.  **`saas-reporting`** : Décisionnel.
6.  **`saas-api`** : Exposition REST et Bootstrap.

Ce découpage garantit l'indépendance des cycles de vie et empêche les dépendances cycliques.

## 4. Règles de Versioning

Nous appliquons **Semantic Versioning 2.0.0** (`MAJOR.MINOR.PATCH`).

### 4.1. Format
`X.Y.Z` (ex: `1.2.0`)

*   **MAJOR (X)** : Changements cassants (Breaking Changes).
*   **MINOR (Y)** : Nouvelles fonctionnalités rétro-compatibles.
*   **PATCH (Z)** : Corrections de bugs rétro-compatibles.

### 4.2. Cycle de Vie (Git Flow simplifié)

*   **Branche `main`** : Contient uniquement les versions stables (Tags).
*   **Branche `develop`** : Branche d'intégration principale (`SNAPSHOT`).
*   **Branches `feature/*`** : Pour les développements en cours.

Les versions en cours de développement portent le suffixe `-SNAPSHOT` (ex: `1.0.0-SNAPSHOT`).
